## 今日课程 / Today's Curriculum

📖 **Handbook 章节**: [氛围编程（Vibe Coding）](https://aiweb3.school/zh/handbook/ai/vibe-coding/) (约 7 分钟阅读，2203 字)

关联章节: [框架（Frameworks）](https://aiweb3.school/zh/handbook/ai/frameworks/) — Vibe Coding 是 Frameworks 思想的实战应用：你需要先理解工作流边界，再决定怎么让 Agent 参与

### 核心知识节点

**1. Vibe Coding 的本质**
- 不是"把需求丢给 AI 然后等代码出现"，而是**人和 AI Coding Agent 共同迭代软件的工作方式**
- 人负责：方向、约束、验收
- Agent 负责：生成、修改、搜索、执行一部分工程动作
- ⚠️ 最大的误解：Vibe Coding = 凭感觉写代码。实际正相反：你要更清楚地管理 repo、任务、上下文、测试和提交边界

**2. 反馈循环的加速与风险**
- 第一性原理：AI Coding 的核心不是自动生成代码，而是**把工程反馈循环压短**
- 过去：搜索文件 → 理解模块 → 改代码 → 跑测试 → 看错误 → 再修改
- 现在：Agent 可以并行承担大部分步骤
- ⚠️ 但如果反馈循环里没有测试、审查和版本控制，速度只会**放大混乱**
- 三条铁律：**任务要小**（越具体产出越可审查）、**上下文要准**（让 Agent 看对文件比写长篇需求重要）、**验证要硬**（跑测试/构建 > "看起来对"）

**3. Claude Code / Codex CLI**
- 这些工具把 AI Agent 放进本地开发环境，让它可以读文件、编辑文件、运行命令
- 关键价值不是"聊天"，而是**能在 repo 里行动**
- 使用前必须明确：哪些文件可以改、哪些命令可以跑、是否允许装依赖、是否允许访问网络、什么时候必须停下来让人确认

**4. Model / API Config**
- 不只是"哪个模型最强"，还要管理：API key、上下文窗口、工具权限、代理设置、成本和日志
- ⚠️ 常见坑：本地和 CI 用的模型不一致、API key 泄漏到日志或提交、上下文太长成本失控

**5. GitHub / gh**
- AI Coding 工作流的协作边界——版本控制仍然是人类审查的关键线
- 实用原则：让 Agent 多做局部 patch，少做不可追踪的大改动
- 每次改动后至少看：git diff、修改文件列表、测试结果、是否含密钥或构建产物

**6. Repo Workflow（高级）**
- 把 AI Coding 放进完整工程流程：issue → branch → commit → test → review → merge → release
- AI 不应绕过这些流程，而是参与其中：从 issue 提炼边界、搜索文件、生成最小 patch、跑测试、写 PR summary

### 第一性原理 / First Principles

> Vibe Coding 的核心不是自动生成代码，而是把工程反馈循环压短。如果反馈循环里没有测试、审查和版本控制，速度只会放大混乱。

- ✅ 适合：搭原型、修小 bug、补测试、写脚本、重构局部模块、解释陌生代码
- ❌ 不适合：无边界地"重写整个项目"——Agent 很容易改到不该改的地方
- ⚠️ 对 BD/产品方向的同学：Vibe Coding 是你理解"AI 能做到什么程度"的最佳窗口——亲自试一次就知道什么是可控的、什么是不可控的

### 在 AI x Web3 中的位置

> AI x Web3 项目通常同时有前端、合约、脚本、索引器、Agent 后端和文档。Vibe Coding 可以显著加快探索速度，尤其适合 hackathon 和早期原型。但链上相关代码的风险更高——合约、签名、权限、支付、迁移脚本不能只靠 Agent 生成后直接上线。

**BD/产品视角的关键思考：**
- Vibe Coding 对非开发者的价值不是"学写代码"，而是**理解 AI Coding 的能力边界**——知道 Agent 能做什么、不能做什么，才能更好地在 BD 合作中评估技术方案
- 合约/签名/支付的代码风险高——作为 BD/产品角色，你要知道**哪些代码需要人工审查、哪部分可以信任 Agent 生成**
- Vibe Coding 让 MVP 速度大幅提升——对 BD 来说，这意味着更快验证产品想法、更快出 demo、更快跟技术团队对齐
- 你自己的 Hermes Agent（我，Cendrine）其实就是 Vibe Coding 思想的体现——我通过 skill 文件 + tool calling 完成你的学习管理，这也是一种"人与 Agent 协作"的实践

## 学习笔记 / Notes

### Vibe Coding 第一性原理

> Vibe Coding 的核心不是自动生成代码，而是把工程反馈循环压短。

- ✅ 优势：原型快速验证、小 bug 修复、脚本生成、陌生代码解释
- ❌ 局限：不能替人做工程判断、不能保证输出安全、不适合无边界重写
- ⚠️ 对 BD/产品方向：理解 AI Coding 的能力边界，比学会写代码更重要

### 人与 AI Coding Agent 的分工

```
人（目标/方向）          Agent（执行/搜索）
     │                        │
     ├── 任务定义    ────────→ ├── 搜索相关代码
     ├── 边界约束    ────────→ ├── 生成修改方案
     ├── 验收标准    ────────→ ├── 执行 patch
     │                        ├── 跑测试/验证
     │                        │
     ←──── 审查 git diff / 测试反馈 ────
     ←──── 迭代或确认 ────────────
```

### "Vibe Coding 铁三角"——每条的 BD 产品解读

| 铁律 | 工程含义 | BD/产品含义 |
|------|---------|------------|
| **任务要小** | 越具体越容易出可审查结果 | 在提需求时，把大目标拆成可执行的小任务 |
| **上下文要准** | 给 Agent 看对文件比写长篇需求更重要 | 好的 brief 要告诉 Agent 背景文件而不是长篇描述 |
| **验证要硬** | 跑测试/构建 > "看起来对" | 让 Agent 提供可验证的产出（截图、URL、数据），而不是"我觉得可以" |

### 链上代码的特殊风险

你的方向是 BD/活动策划/Hackathon——当项目中涉及链上操作时：

| 代码类型 | Agent 可做 | 必须人工确认 |
|---------|-----------|------------|
| 合约代码 | 写测试、解释 ABI、生成脚手架 | 部署前的安全审查、Gas 估算 |
| 签名逻辑 | 生成签名模板 | 签名内容、权限范围 |
| 支付脚本 | 生成交易模拟 | 实际 Gas 花费、收款地址 |
| 迁移脚本 | 生成脚本草案 | dry-run 结果、数据检查 |
