### 核心知识节点

**1. RPC Tool — 最基础的链上读取入口**
- Agent 通过 RPC 读取链状态、查询区块、估算 gas、获取日志
- ⚠️ **读写必须分离**：只读 RPC 工具和广播交易的 RPC 工具不能混在一起
- 工具返回应包含：chain id、RPC provider、block number、method、result、error——这样 Agent 才能说明数据来源

**2. Contract Read vs Contract Write — 风险分层的核心分界**
- **Contract Read**（初级）：读余额、配置、owner、allowance、pool 状态——最常用、相对低风险
- **Contract Write**（高级）：改变链上状态——需要链 ID 检查、地址验证、ABI 匹配、gas 估算、simulation、policy 检查、用户授权、交易回执追踪
- ⚠️ 读错网络或读错合约，模型会基于错误事实给出建议——**数据来源可追溯比数据本身更重要**

**3. Wallet Tool — 最敏感的边界**
- 必须把「连接」「签名消息」「发送交易」「授权 token」「撤销授权」分成不同动作
- AI 生成的交易草稿**不应绕过钱包确认**
- 高风险动作必须回到用户、Smart Account policy 或多签流程

**4. DeFi Tool — 直接影响资产的最危险层**
- 至少需要：协议白名单、最大交易额、最大滑点、价格来源、simulation、allowance 检查、人工确认或 session key policy
- ❌ **永远不要给 Agent 一个"帮我调用任意 DeFi 协议"的通用工具**

**5. Tool Permission & Tool Log — 治理和审计两条生命线**
- 权限分层：查询自动允许 / 生成交易草稿自动允许 / 小额白名单 session key / 大额人工确认 / 任意合约默认禁止
- 日志必须记录：用户目标、工具名、输入输出、错误、时间、链 ID、区块号、交易哈希、确认人、policy 判断

### 第一性原理 / First Principles

> **模型可以选择工具，但工具必须用确定性边界限制模型。** —— Web3 Tool Use 第一性原理
> **不要让 Agent 直接拼接任意 calldata 或调用任意地址。工具应该把危险能力封装成受限接口。**

| 核心原则 | 含义 |
|---------|------|
| ✅ 读写分离 | 读取链上状态和发送交易必须是不同工具、不同权限 |
| ✅ 参数结构化 | chain id、contract address、method、args 不能埋在自然语言里 |
| ✅ 日志不可省 | 每次工具调用都要记录输入、输出、时间、来源和错误 |
| ✅ 逐层收紧权限 | 从读取到 DeFi 执行，每多一层风险就多一层确认 |

### 在 AI x Web3 中的位置

> Web3 Tool Use 是从"AI 能解释链上信息"走向"AI 能参与链上执行"的关键层。它连接 Chain-aware Context、Agent Workflow 和 Agent Wallet。

**BD/产品视角的关键思考：**
- 你作为 BD/产品方向，Web3 Tool Use 的核心问题不是"怎么实现"，而是**「哪些操作应该让 AI 自动完成，哪些必须保持人工确认」**——这就是产品设计中的权限分层决策
- 做活动策划时，可以把 Tool Permission 的分层思路用在活动协作流程上：读活动信息谁都能做、报名需要确认、领取奖励需要签名——和 Web3 工具调用的权限模型完全相通
- DeFi Tool 的「白名单协议 + 额度限制 + simulation + 确认」四步框架，可以直接用在产品风险评估中：合作方谁在权限白名单里？合作金额有没有上限？能不能先模拟再执行？
- Web3 Tool Use 对你最大的启发是：**读（research）和写（transact）在产品和协议层面必须分开设计**——一个用户能看市场分析不等于她授权了自动交易

## 学习笔记 / Notes

### Web3 Tool Use 第一性原理

> 模型可以选择工具，但工具必须用确定性边界限制模型。

- ✅ AI Agent"能调用"Web3 工具不难——难的是权限、参数、模拟和日志
- ❌ 不要让 Agent 直接拼接任意 calldata 或调用任意地址
- ⚠️ BD/产品方向：你不需要实现工具，但必须能判断"这个操作应该让 AI 做还是让人做"

### Web3 工具风险分层矩阵

```
┌─────────────────────────────────────────────────────┐
│                  RISK LADDER                         │
│                                                     │
│  🟢 RPC Tool ───── 只读链状态、查询区块               │
│  🟢 Contract Read ── 读余额、配置、view 函数          │
│  🟡 Explorer Tool ── 查交易、合约源码、event          │
│  🟠 Wallet Tool ──── 连接、签名、发送交易（需确认）    │
│  🔴 Contract Write ─ 改变链状态（需 simulation + 授权）│
│  🔴 DeFi Tool ────── swap/借贷/仓位（白名单+额度限制） │
└─────────────────────────────────────────────────────┘
     ↓
  权限要求逐层递增
```

### BD/产品参考：Web3 Tool Permission vs 活动策划权限

| 风险等级 | Web3 Tool Use 对应 | 活动策划对应 |
|---------|-------------------|------------|
| 自动允许 | RPC / Contract Read | 浏览活动列表、查看日程 |
| Session Key | 小额白名单支付 | 快速报名（白名单用户） |
| 需人工确认 | Contract Write / Wallet Sign | 大额赞助、合约签约 |
| 默认禁止 | 任意合约调用 / 任意 DeFi | 随意修改活动规则、直接发起转账 |

**[EN-progressive — Day 9 — Mostly English with CN annotations]**

📚 **Day 9 — Web3 Tool Use: Permission Boundaries for AI Agents on Chain**

Today marks the transition from AI Basics into the AI×Web3 Bridge section. Web3 Tool Use is about giving AI agents structured, permissioned access to on-chain operations — from reading balances and checking contract states, all the way to generating transaction drafts and requesting wallet signatures.

Key concepts:
- **Read/Write Separation** — read-state RPC and write-transaction tools must be different tools with different permissions
- **Tool Permission** — permission levels from auto-allow (balance queries) through session key (small payments) to human-confirmed (large DeFi operations) to default-deny (arbitrary contract calls)
- **Tool Log** — every agent call must record: goal, tool name, inputs, outputs, error, chain ID, tx hash, and who approved

From a BD/Product perspective: The risk ladder in Web3 Tool Use (RPC → Contract Read → Wallet → Contract Write → DeFi) maps directly to product design decisions — "what should the AI do autonomously vs. what requires human confirmation." This framework is surprisingly applicable beyond Web3, like event planning workflows and partnership risk assessment.

> "The hardest part of Web3 Tool Use is not getting the AI to call a contract — it's designing the permission model that determines what it's allowed to do."
