## 今日课程 / Today's Curriculum

📖 **Handbook 章节**: [模型上下文协议（MCP）](https://aiweb3.school/zh/handbook/ai/mcp/) (约 5 分钟阅读，1408 字)

关联章节: [智能体（Agent）](https://aiweb3.school/zh/handbook/ai/agent/) — Agent 的工具调用能力必须和状态、权限、停止条件一起设计，MCP 是这层的标准化方案

### 核心知识节点

**1. MCP 的本质：工具接入的标准化协议**
- MCP 不是让模型更聪明，而是让模型以**可描述、可复用、可管理的方式使用外部能力**
- 类比：MCP 之于 AI 工具调用 ≈ USB 之于外设——定义了接口标准，而不是每个设备自己发明接线方式
- ⚠️ MCP 的核心价值不是"能接更多工具"，而是让工具接入变得**可描述（schema）、可发现（server 暴露能力）、可限制（权限模型）**

**2. Server 与 Client 的分工**
- **MCP Server**：提供能力的一侧，暴露 resources（资源）、tools（工具）、prompts（提示模板）
- **MCP Client**：连接模型和 Server 的一侧（桌面应用、IDE、Agent runtime、聊天客户端）
- ⚠️ 不加限制的 Server 容易把本地文件、内部 API 或高风险动作暴露给模型——**边界设计是 Server 的第一要务**

**3. Tool Schema**
- 描述工具名字、用途、参数、返回值和约束——模型正确调用工具的关键
- 好的 schema 不只是字段类型，还要说明：什么时候用、参数含义、哪些字段必填、是否修改外部状态、失败时返回什么
- ⚠️ Tool schema 写得模糊，模型就会用错误参数填空

**4. Permission（权限）—— MCP 最容易被低估的问题**
- 读文档、查 issue、创建 PR、发支付、删文件——这些动作风险完全不同
- 至少需要区分：只读 vs 写入、会话授权 vs 长期授权、是否需要用户确认、是否可撤销和审计
- ⚠️ 方便不等于安全——MCP 让工具连接更方便，但权限模型必须独立实现

### 第一性原理 / First Principles

> **模型不应该直接拥有世界；它应该通过明确协议访问被授权的上下文和工具。** —— MCP 第一性原理

| 核心原则 | 含义 |
|---------|------|
| ✅ 工具要有 schema | 没有 schema，模型调用工具就会变成自然语言猜参数 |
| ✅ 权限要在协议外也成立 | 协议能描述能力，但真正的授权、审计和隔离仍要由系统实现 |
| ✅ 错误要可传递 | 工具失败、超时、权限不足，必须明确返回，而不是让模型猜 |

### 在 AI x Web3 中的位置

> MCP 在 AI x Web3 中可以作为 Agent 连接链上工具和开发工具的接口层。例如读取区块浏览器、查询合约文档、调用 RPC、生成交易草稿、读取项目 issue。但 MCP 本身不是钱包安全方案。它可以标准化工具入口，不能替代账户权限、交易模拟、签名确认、session key 和审计日志。比较稳的设计是：MCP 负责工具发现和调用格式；Web3 账户系统负责最终权限和执行边界。

**BD/产品视角的关键思考：**
- MCP 对你来说最重要的概念是**权限分层**——在 BD 合作中，当技术团队说"我们接入了 MCP"，你要追问的是：哪些工具暴露了？写权限是谁控制的？调用有没有审计日志？
- 做活动策划/Hackathon 时，MCP 的标准化思路可以用在**活动流程设计**上——把每个环节（报名→签到→抽奖→领奖）定义成"工具"并明确输入输出，活动执行会清晰很多
- MCP 的「只读 vs 写入」分类框架可以直接用在产品设计中：哪些信息用户不需要确认就能读？哪些操作（支付、合约调用）必须走审批流程？
- MCP 本身是纯基础设施层——**它对 BD/产品的价值不是去实现它，而是理解它的边界**，这样在项目讨论中能正确判断"哪个环节该用 MCP，哪个环节不该用"

│  - 生成调用请求    │      │  - 处理用户授权提示     │      │  └────────┘  │
└─────────────────┘      └──────────────────────┘      └──────────────┘

                        ┌──────┴──────┐

### 你的 Hermes Agent = MCP Client 实战实例

你已经在用 MCP 的核心理念了——我就是你的 MCP Client：

| MCP 概念 | Hermes Agent 中的体现 | 你的控制 |
|---------|---------------------|---------|
| Server 暴露工具 | 我通过 tool calling 访问文件、搜索、网页、浏览器 | 你控制哪些工具调用权限打开 |

**[中英渐进 — Day 8 — 逐步增加英文比例]**

📚 **Day 8 — Model Context Protocol (MCP): Standardizing How AI Agents Connect to the World**

MCP 解决了 AI 应用生态中的一个关键问题：模型需要读文件、查数据库、调用 API、操作工具，但每个应用都用自己的方式接入，生态非常碎片。MCP 的价值是把工具接入标准化——这不是让模型更聪明，而是让模型以可描述、可复用、可管理的方式使用外部能力。

Key concepts:
- **MCP Server / Client** — standardized protocol for tool discovery and invocation
- **Tool Schema** — clearly defined inputs, outputs, and side effects so models don't guess parameters
- **Permission** — the most underestimated layer: read vs write, session vs permanent, auditable

From a BD/Product perspective: MCP's permission layer framework (read-only → write-reversible → write-high-risk) is surprisingly practical for designing any AI-integrated product. The question "who controls write access and who audits the logs" applies whether you're building an agent wallet or a Telegram event bot.

> "MCP standardizes the interface — but the permission model determines whether that interface is useful or dangerous. In AI×Web3, both matter equally."
