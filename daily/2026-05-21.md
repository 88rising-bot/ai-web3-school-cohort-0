### 核心知识节点

**1. Instruction（任务指令）**
- Prompt 不是"神奇咒语"，而是把模糊任务变成模型可稳定执行的工作说明
- 实用四段式：**任务目标** → **可用输入** → **禁止行为** → **输出格式和失败格式**
- 关键区分：解释 vs 执行 — 研究助手可以整理资料，但不能假装结论已被验证

**2. Few-shot（少样本示例）**
- 在 prompt 里放少量示例，让模型模仿判断方式和输出格式
- 适合"风格和边界很难一句话说清"的任务
- ⚠️ 维护成本：协议升级、字段变化后，旧示例可能反过来误导模型
- 示例不是一次写完的素材，而是要跟 eval 一起维护的测试资产

**3. Structured Output（结构化输出）**
- 让模型按固定结构返回结果（JSON schema、函数参数等）
- 不是为了好看，而是为了让后续代码能检查、拒绝、记录和回归测试
- 范例字段: `action`, `risk_level`, `requires_human_approval`, `uncertainties`

**4. Prompt Injection（提示注入）**
- 攻击者通过用户输入/网页/文档让模型忽略原始规则、泄露信息或调用危险工具
- Agent 场景尤其危险 — 模型不只是回答问题，还能调用工具、触发外部动作
- 应对方案：外部内容标记为不可信 + 工具参数校验 + 敏感动作强制 allowlist + human approval

### 在 AI x Web3 中的位置

> "Prompt 定义任务和输出格式。Context 提供可信数据和来源边界。Model 生成解释或候选动作。Code 校验 schema 和业务规则。Guard / simulation 检查链上影响。Human check 确认高风险动作。"
> — Prompt 不独自承担安全，它是安全链路的第一层

**BD/产品视角的关键思考：**
- Prompt 是产品的"沟通协议" — 决定了 AI 能理解什么、不能做什么
- 对 BD 来说：好的 prompt 可以让 AI 稳定输出竞品分析、活动策划案、项目调研报告
- 但 prompt 不能替代事实校验 — 链上数据和官方文档仍需要二次确认
- Structured Output 让 AI 输出可以被下游系统直接消费，这对自动化 workflow 很重要

## 学习笔记 / Notes

### Prompt 第一性原理

> "Prompt 是软约束，不是安全边界。真正的边界必须由代码、权限、校验和审计来承担。"

- ✅ Prompt 擅长：表达任务目标、规范输出格式、控制模型行为风格
- ❌ Prompt 不能：替代权限检查、阻止注入攻击、保证事实准确
- ⚠️ 对 BD 来说：用 prompt 让 AI 输出结构化内容（表格/JSON）远比让 AI "自由发挥"更可靠

### Instruction 四段式写法

| 段落 | 内容 | 例子 |
|------|------|------|
| 任务目标 | 你是什么角色、要完成什么 | "你是一个 Web3 市场分析师" |
| 可用输入 | 哪些信息可以使用 | "使用下面提供的协议数据" |
| 禁止行为 | 不能做什么 | "不要编造不存在的数据" |
| 输出格式 | 结构和失败格式 | "输出 JSON: {summary, risk_level}" |

### Context Engineering 层级

```
指令层：系统规则、工具使用规则、禁止事项
任务层：用户目标、本次会话参数
事实层：链上状态、工具结果、simulation
知识层：文档、标准、论坛、历史案例
记忆层：用户偏好和项目配置
```

### BD 场景的 Prompt 实践

**活动策划 Prompt 示例：**
```
你是一个 Web3 活动策划顾问。
根据以下项目信息，输出一个活动方案框架（JSON格式）：
- 活动名称
- 目标受众
- 核心环节（3-5个）
- 预期参与人数
- 风险点及应对
- 推荐的 KOL/合作伙伴类型
禁止编造数据。
```
