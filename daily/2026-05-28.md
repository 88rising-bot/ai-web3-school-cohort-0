### 核心知识节点

> Agent Workflow 的核心，是把概率模型放进确定性流程里。

**1. Task Graph（任务图）— 中级**
- 把目标拆成节点和依赖，而不是让 Agent 一口气自由执行
- 例：ERC-20 swap 拆成 9 步：读取目标 → 获取余额 → 查价格 → 生成候选 → 模拟 → 展示风险 → 用户确认 → 发送 → 追踪
- 每一步都可设：输入、输出、权限、停止条件

**2. State Machine（状态机）— 高级**
- 链上工作流常见状态：`draft` → `context_loaded` → `plan_ready` → `simulation_failed` → `waiting_user_confirmation` → `submitted` → `confirmed` → `reverted` → `cancelled`
- ⚠️ 状态机的价值：用户刷新页面、交易 pending、RPC 失败、模型重试时，系统不会忘记自己在哪里，也不会重复执行危险动作

**3. Human-in-the-loop（人机协同确认）— 中级**
- 合理分层：只读分析→自动 | 交易草稿→自动生成 | 小额白名单→session key | 高风险→人工确认 | 超出 policy→直接拒绝
- ⚠️ 重点不是「有没有人工确认」，而是人确认时能否看懂资产变化和失败风险

**4. Retry / Fallback（重试与降级）— 中级**
- Web3 Agent 不能盲目重试：发送交易失败要先判断是否已广播，交易 pending 不能简单再发一笔
- Fallback 要谨慎：模型不可用可降级只读，但不该自动换未评估的模型继续发交易

**5. Trace（追踪）— 初级**
- 至少包括：用户目标、模型版本、上下文来源、工具输入输出、policy 判断、simulation 结果、人工确认、交易哈希和最终状态
- 没有 trace，出了问 题只能看聊天记录；有 trace，才能复盘是哪一步出的问题

**6. Evaluation Harness（评估框架）& Regression Set（回归集）— 高级/中级**
- 对链上 Agent，eval 不只要测回答好不好，还要测：是否正确拒绝越权请求、是否识别错误链、是否要求 human check
- Regression Set 是固定测试用例池，正常/错误链/滑点过大/余额不足/用户拒绝各一种

### 第一性原理 / First Principles

> **高风险 Agent 不能只有「下一步推理」，必须有状态、边界和停止条件。**
> **模型可以规划，但系统要知道当前到了哪一步、哪些工具已调用、哪些结果可信、哪些动作需要确认、失败后如何停止。**

| 核心原则 | 含义 |
|---------|------|
| ✅ 流程要显式 | 不要把完整执行链路藏在一段长 prompt 里 |
| ✅ 状态要可恢复 | 工具失败、用户拒绝、交易 pending 时，系统要知道如何继续或停止 |
| ✅ 评估要可回放 | 没有 trace 和 regression set，很难知道改模型后是否更安全 |

### 在 AI x Web3 中的位置

> Agent Workflow 是桥接层的**流程骨架**。Chain-aware Context 提供事实，Web3 Tool Use 提供能力，Agent Wallet 提供权限边界，Workflow 把它们组织成可执行但可控的路径。

**BD/产品视角的关键思考：**
- Agent Workflow 对你最重要的概念是 **Task Graph**——它本质上是产品流程图：把用户需求拆解成每一步的输入、输出、谁（人或 AI）执行、什么情况下停止。做活动策划时也完全可以用这个思路：报名→资格验证→支付→签到→抽奖→领取奖励，每一步的「谁负责」「失败怎么办」
- **Human-in-the-loop** 的分层理念可以直接用在产品设计中：哪些操作让用户自己完成，哪些交给 AI，哪些必须两个人签字确认
- 做 BD 合作时，问技术团队「你们 Agent 的 workflow 有 trace 吗？出了错怎么回溯？」比问「你们用的什么模型」更能判断项目成熟度
- **Retry/Fallback** 的「不能盲目重试」原则，在商务合作中同样成立：合同发送失败不能简单重发，要先查是否已经发送成功——和 Web3 Agent 的 retry 逻辑完全一致

### BD/产品参考：Task Graph → 活动策划工作流对比

| Agent Workflow 概念 | 活动策划对应 |
|--------------------|------------|
| Task Graph | 活动甘特图/节点拆解：预热→报名→执行→收尾 |
| State Machine | 活动状态：draft→审核中→已发布→名额已满→已结束→已取消 |
| Human-in-the-loop | 大额赞助需要审批，小额报名自动通过 |
| Retry/Fallback | 系统发送活动提醒失败→先查是否已发送→再决定是否重试 |
| Trace | 活动运营日志：谁做了什么、什么时候、结果是什么 |
| Regression Set | 活动应急预案：报名人数超标、嘉宾临时取消、技术故障 |

**[EN-progressive — Day 10 — Mostly English with CN annotations]**

📚 **Day 10 — Agent Workflow: Turning Probabilistic Models into Deterministic Processes**

Today's focus is Agent Workflow — the structural layer that organizes AI capabilities (context reading, tool calling, wallet interaction) into a controlled, auditable execution chain. Without a well-designed workflow, an AI×Web3 project risks being "fast in demos but dangerous with real assets."

Key concepts:
- **Task Graph** — break a goal into nodes with clear inputs, outputs, permissions, and stop conditions. Each node is a decision point, not just a step.
- **State Machine** — explicit states (draft → plan_ready → waiting_user_confirmation → submitted → confirmed → reverted) prevent the system from forgetting where it is, even after failures or refreshes.
- **Human-in-the-loop** — place humans at critical risk points, not every low-risk step. The real challenge: can the human actually understand what they're approving?
- **Retry / Fallback** — never blindly retry a failed on-chain write. First check if the transaction was already broadcast.
- **Trace + Regression Set** — without audit trails and safety test cases, you can't know if a model update made the system safer or more dangerous.

From a BD/Product perspective: Agent Workflow's Task Graph is essentially a **product flowchart** for AI-native systems. Every product manager who's designed a signup→verify→pay→confirm flow already understands the core idea — the difference is that in AI×Web3, each node needs explicit permission boundaries, failure recovery, and audit documentation.

**Practice idea:** Draw a Task Graph for your own Learning Agent's morning routine — what are the nodes, who (AI vs human) executes each, and where are the failure points? That's Agent Workflow in action.
