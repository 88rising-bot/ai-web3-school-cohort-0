### 核心知识节点

**1. Chunking（文档切片）**
- RAG 的第一步：把长文档切成可检索的片段
- 切太小→语义断裂；切太大→噪声多、token 成本高
- 最佳实践：按结构切（标题、API endpoint、函数说明、FAQ），每个 chunk 保留来源 URL、标题路径、更新时间
- ⚠️ 技术文档尤其小心：参数表、风险提示、示例代码常跨段落，不能按固定字数硬切

**2. Vector DB（向量数据库）**
- 存储 embedding，按语义相似度检索相关 chunk
- ⚠️ 向量相似 ≠ 答案正确 — 旧版本文档可能与问题"高度相似"但不适用
- 关键：metadata 过滤（来源、版本、chain、更新时间、可信等级、是否废弃）

**3. Retriever（检索器）**
- 取回候选材料的组件：向量检索 / 关键词 / 混合检索 / 图检索
- 不能只看语义相似度 → 还要知道版本、环境、时间、产品线
- 实用判断：如果用户问题含版本/时间/地址，就不能只做纯向量搜索

**4. Rerank（重排序）**
- 初步检索后重新排序：把更相关、更可信、更完整的内容排到前面
- ⚠️ 增加延迟和成本 → 资产风险场景值得加，小型 FAQ 未必需要

**5. Citation（引用溯源）**
- 把答案的关键结论连接回来源 → 不是装饰，是用户验证入口
- 至少说明：来源文档、是否官方、版本/更新时间、模型归纳 vs 有证据
- 没有 citation 和 freshness 的 RAG = 把幻觉从模型内部搬到了检索系统

### 第一性原理

> RAG 的可靠性取决于**证据链**，不取决于向量库这个名词。

一个 RAG 系统至少有**三次判断**：
1. 文档怎么切（Chunking）
2. 查询时取哪些内容（Retriever + Rerank）
3. 生成时如何引用和拒答（Citation）

任何一层做错，模型都会拿着**错误材料说得很顺**。

三条原则：
- ✅ 检索结果不是事实，只是候选证据
- ✅ 引用要能回到原文
- ✅ 检索失败要允许拒答（"不确定"）

### 在 AI x Web3 中的位置

RAG 位在 Knowledge Base 和模型之间。它帮 Agent 查资料、补上下文、引用证据，但不负责最终执行。

**常见应用：**
- 协议文档问答 → BD 可以快速查询项目方文档
- 合约接口解释 → 不用每次去翻源码
- 治理提案和论坛摘要 → 快速了解 DAO 讨论脉络
- 审计报告检索 → 评估项目安全背景
- SDK / API Copilot → 开发者效率工具

**⚠️ 关键红线：** RAG 结果要影响链上动作时，还需接 simulation、policy 和 human check。文档说某函数"可以调用"≠ 当前用户"应该签名"。

### BD/产品视角的思考

RAG 对 BD/产品方向的价值：
- **竞品调研**：RAG 可以帮你在文档、白皮书、论坛中快速检索目标协议的关键信息
- **活动策划**：检索过往活动案例和社区反馈，辅助方案设计
- **项目评估**：快速检索审计报告、治理提案、团队背景，减少人工查阅时间
- **但 RAG 不能替代判断** — 检索到的信息需要结合链上数据和实时状态做交叉验证

## 学习笔记 / Notes

### RAG 第一性原理

> "RAG 的核心不是让回答更长，而是让回答有来源、有版本、有边界。"

- ✅ RAG 擅长：在大量文档中找到相关材料，并基于这些材料生成引用答案
- ❌ RAG 不能：验证检索结果的事实准确性、代替权限检查、保证信息来源是最新的
- ⚠️ 对 BD 来说：RAG 可以大幅减少"这个协议是干什么的"这类调研时间，但对"这个协议值得合作吗"的判断仍需人工

### RAG 五层流程

```
用户提问
    ↓
Chunking — 文档按结构切片，保留 metadata
    ↓
Vector DB — 语义相似度检索 + metadata 过滤
    ↓
Retriever — 取回候选材料（向量/关键词/混合）
    ↓
Rerank — 重新排序（可选，按场景）
    ↓
Generation — 基于检索结果生成回答 + Citation
    ↓
拒答或提示版本/不确定性
```

### Context Engineering 层级（来自 Context 章节）

```
指令层：系统规则、工具使用规则、禁止事项
任务层：用户目标、本次会话参数
事实层：链上状态、工具结果、simulation
知识层：文档、标准、论坛、历史案例
记忆层：用户偏好和项目配置
```

> RAG 主要负责填充"知识层"，并在答案中链接回来源。

### BD 场景的 RAG 实践思路

**协议调研 RAG 的 chunk 设计示例：**
| 文档类型 | chunk 策略 | metadata |
|---------|-----------|---------|
| 白皮书 | 按章节切 | 标题、版本、日期 |
| 技术文档 | 按 API/函数切 | endpoint、版本、 chain |
| 审计报告 | 按发现项切 | 风险等级、日期、审计方 |
| 治理提案 | 按提案 ID 切 | 提案人、状态、投票截止 |

📚 Day 4 — Retrieval-Augmented Generation (RAG)

今天学习了 RAG（检索增强生成）：它不是"给模型接一个向量库"那么简单，而是一条完整的证据链——从文档切片、向量检索、重排序到引用溯源。

关键概念：Chunking（按结构切文档而非固定字数）、Vector DB（存向量+metadata，不只看语义相似度）、Retriever（不能只做纯向量搜索）、Rerank（按场景决定是否值得加延迟成本）、Citation（让答案可验证）。

BD/产品视角的发现：在做协议调研或竞品分析时，RAG 可以帮我快速定位文档中的关键信息，减少翻阅白皮书的时间。但检索到的信息需要结合链上数据和实时状态做交叉验证——RAG 不能替代实地调研。

> "RAG's value isn't making answers longer — it's making answers verifiable. Without citation and freshness, RAG just moves hallucination from the model to the retrieval system."

Next: Agent（智能体）— 让 AI 不只是回答问题，而是自主执行任务。
