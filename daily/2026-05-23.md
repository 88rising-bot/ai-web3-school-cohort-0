### 核心知识节点

**1. Tool Use（工具调用）**
- Agent 调用外部能力：搜索、数据库、API、代码执行、邮件、支付接口等
- 工具让 Agent 从"会回答"变成"能做事" — 但工具也让 Prompt Injection 变得危险
- 一个能读网页的 Agent 被恶意网页影响，和一个能写入系统的 Agent 被恶意网页影响，后果完全不同
- ⚠️ 工具设计要明确：输入 schema、权限范围、是否只读、是否产生外部副作用、调用前后如何记录、哪些调用需人工确认

**2. Planning（任务规划）**
- 把目标拆成步骤 — "分析这个 DAO 提案" → 读取提案、检索讨论、总结争议、检查投票机制、生成风险清单
- ⚠️ 计划只是候选路线，不是授权 — 越靠近高风险动作，计划越需要被系统规则拆开检查
- 好的 Plan 应暴露：每一步需什么工具、读还是写、哪些可自动执行、哪些需用户确认、失败可否重试、最终如何验证

**3. State（状态管理）**
- Agent 当前任务的状态：用户目标、已完成步骤、工具返回、错误、预算、确认记录
- ⚠️ 很多 Demo 把 state 只放在 prompt 历史里 — 生产系统需要可查询、可恢复、可审计的 state
- 有外部执行场景还应记录：环境、版本、关键参数、调用结果、确认请求、撤销事件

**4. Reflection（自我反思）**
- 让 Agent 检查中间结果：信息不足、工具失败、计划不合理 → 修正下一步
- 适合提升复杂任务质量，但⚠️不能替代外部验证 — 模型可能给自己的错误找理由
- 自我检查可以提高质量，**确定性检查才能承载风险**

**5. Multi-Agent（多智能体协作）**
- 多个 Agent 分工协作：研究 Agent 读资料、开发 Agent 写代码、安全 Agent 做审查
- ⚠️ 放大协调问题：上下文传递丢失、责任边界模糊、一个 Agent 的错误被另一个当成事实
- 先问一个朴素问题：多个 Agent 是否真的减少复杂度？

### 第一性原理 / First Principles

> Agent 不是自主权本身，而是被约束的执行循环。目标、工具、状态、权限和停止条件缺一不可。

- ✅ 工具比回答更危险：读数据 ≠ 写数据库 ≠ 发送请求 ≠ 改配置
- ✅ 状态必须外置和可查：任务进度、工具结果、失败原因、用户确认都要记录
- ✅ 停止条件要明确：达到目标、超出预算、信息不足、风险越界、用户拒绝 → 都应让 Agent 停下来

### 在 AI x Web3 中的位置

> Agent 位在模型能力和链上执行之间。它可以把用户目标推进成多步流程，但不能绕过账户、权限和结算规则。

一个相对稳健的 AI × Web3 Agent 架构：
```
用户提出目标和约束
    ↓
Agent 读取上下文并生成计划
    ↓
系统把计划拆成只读步骤和候选写入步骤
    ↓
只读工具自动执行，写入工具进入 policy 检查
    ↓
Simulation 展示链上影响
    ↓
用户确认高风险动作
    ↓
Wallet / Smart Account 执行
    ↓
日志记录每一步和最终状态
```

⚠️ 最危险的设计：让 Agent 同时拥有模糊目标、广泛工具、长期记忆和大额资产权限。

**BD/产品视角的关键思考：**
- Agent 对 BD 的价值：可以设计"协议调研 Agent"自动收集项目数据、生成分析报告
- 对活动策划：可以设计"活动策划 Agent"辅助方案生成、预算规划、KOL 匹配
- 但 Agent 不能替代 BD 的判断力 — 商务决策仍需要人工基于信任关系和市场直觉做出
- 产品层面：Agent 的输出格式（结构化 JSON）让后续工作流更容易集成

## 学习笔记 / Notes

### Agent 第一性原理

> "模型负责提出候选行动，系统负责限制行动空间，用户负责批准高风险边界。"

- ✅ Agent 擅长：多步任务拆解、工具调用、信息收集、流程自动化
- ❌ Agent 不能：承担安全责任、绕过权限检查、替代人工决策
- ⚠️ 对 BD 来说：Agent 可以自动做"调研"但不能自动做"判断"

### Agent 执行循环

```
[目标] → [Plan] → [Tool Call] → [观察结果] → [反思] → [下一步] → [完成/停止]
                                     ↓
                            State 持久化记录每一步
```

### 工具风险分级（BD/产品视角）

| 工具类型 | 风险等级 | BD 场景 | 是否需要人工确认 |
|---------|---------|---------|---------------|
| 搜索/读取 | 🟢 低 | 竞品信息检索 | 否 |
| 数据库查询 | 🟢 低 | 历史活动数据 | 否 |
| 内容生成（文档/邮件） | 🟡 中 | 活动方案初稿 | 建议审阅 |
| API 调用（只读） | 🟡 中 | 链上数据查询 | 否 |
| 写入数据库 | 🔴 高 | 更新活动记录 | 是 |
| 发送邮件/消息 | 🔴 高 | 联系 KOL/合作伙伴 | 是 |
| 支付/签名 | ⛔ 极高 | 活动预算支出 | 必须人工 |

### Context 与 Agent 的关系（补读 Context 章节的要点）

Context Engineering 五层结构：
```
指令层：系统规则、工具使用规则、禁止事项
任务层：用户目标、本次会话参数
事实层：链上状态、工具结果、simulation
知识层：文档、标准、论坛、历史案例
记忆层：用户偏好和项目配置
```

Agent 的 Planning + Tool Use 跨越多层 Context — 理解分层有助于设计更稳健的 Agent 行为。

📚 Day 5 — Agent: From Answering to Executing

今天学习了 Agent（智能体）：它不只是"更聪明的问答模型"，而是一个被约束的执行循环——能拆任务、调工具、读状态、反思修正，但每一步都在权限边界内运行。

关键概念：Tool Use（工具让 Agent 从回答变成做事）、Planning（计划只是候选路线不是授权）、State（状态不能只在 prompt 历史里必须外置可查）、Reflection（自我检查不是安全边界）、Multi-Agent（多个 Agent 未必更好）。

BD/产品视角的发现：Agent 可以帮我做协议调研、竞品分析和活动策划的初稿生成。但商务决策仍然需要人工判断——Agent 可以做调研、出方案，但它没有信任关系和市场直觉。

> "Agent boundaries — what it cannot do — define its safety more than what it can do. In Web3, the most dangerous design is an agent with vague goals, broad tools, long memory, and large asset permissions."
