### 核心知识节点

> Agent Wallet 不是"给 AI 一个钱包私钥"这么简单。真正要解决的问题是：Agent 可以代表用户做哪些链上动作，这些动作有没有额度、时间、对象和风险限制，用户什么时候必须重新确认。

**1. AA Wallet / Smart Account — 中级**
- AA Wallet 的关键不是"钱包更高级"，而是**账户终于可以表达规则**：谁能操作、能操作什么、什么时候失效、超出边界怎么办
- Smart Account 是 Agent Wallet 的执行边界，用合约规则承载权限、恢复和自动化
- ⚠️ 判断一个 Agent Wallet 是否靠谱，重点看它是不是把"授权给 Agent"拆成了**可检查的规则**，而不是一句模糊的"允许自动操作"

**2. Session Key（会话密钥）— 高级**
- 把临时、有限权限的小钥匙给 Agent，而非主私钥
- 必须限制：有效时间、可调用的合约或方法、单笔金额和总额度、可操作的 token 类型、是否允许转出到外部地址、可随时撤销
- ⚠️ Session Key 不是小号私钥，而是一组**受限能力**

**3. Policy（策略）— 高级**
- Policy 把"Agent 可以做什么"写成系统能检查的规则
- 好的 policy 应该让系统能回答：**这笔操作到底有没有越界？**
- 模糊的"帮我低风险操作一下"不适合直接变成链上权限

**4. Guard（卫士）— 高级**
- 执行前的确定性检查层，Agent 生成候选动作，Guard **必须**拒绝越界动作
- 即使 Agent 因模型误判/工具错误/Prompt Injection 产生不合理的交易，Guard 负责最后拦截

**5. Simulation（模拟）— 中级**
- 在交易发出前预演结果，把模拟结果翻译成人能看懂的内容
- 不只是技术检查，也是**用户理解和确认的入口**
- 要回答：这笔交易会让用户失去什么、得到什么、承担什么风险

**6. Revocation（撤销）— 中级**
- 比"怎么授权"更重要的是"**怎么收回**"
- 撤销不应只靠用户手动操作：session key 到期、额度用完、多次失败、异常地址、偏离任务——都可以自动收紧权限
- 一个简单的产品原则：**所有自动化权限，都应该有一个用户看得见的关闭入口**

**7. Human Check（人机确认）— 初级**
- 分层：低风险→自动 | 中等→模拟后确认 | 高风险→明确展示影响并要求确认 | 超出 policy→直接拒绝
- 好的 Human Check 应该让用户看懂自己在批准什么，而不是只看到一串哈希和"Confirm"按钮

### 第一性原理 / First Principles

> **控制权不能交给一个概率系统。Agent 只能拿到可验证、可限制、可撤销的行动空间。**

| 核心原则 | 含义 |
|---------|------|
| ✅ Agent 不直接控制主资产 | 模型可以生成计划和交易草稿，但不应该拿到用户主私钥。高风险动作交回给人确认 |
| ✅ 钱包不只是签名按钮，而是权限系统 | 表达时间、金额、合约、方法、资产、接收方和撤销条件，让模糊授权变成可检查规则 |
| ✅ 自动化必须绑定撤销能力 | 用户随时要能看见 Agent 拿了什么权限、做过什么、还能做什么，以及在哪里关掉它 |

### 在 AI x Web3 中的位置

> Agent Wallet 处在 AI x Web3 的关键连接点上。AI 负责理解目标、整理上下文、生成计划；Web3 钱包负责身份、资产、签名和链上执行。Agent Wallet 把这两边接起来，但不能让模型直接越过用户控制权。

**BD/产品视角的关键思考：**
- Agent Wallet 的 **Policy + Guard** 概念对产品设计的启发：任何 AI 功能上线前，都应该问"Agent 的边界在哪里？越界了谁拦住？"——产品经理需要定义的不是"AI 能做什么"，而是"AI **不能**做什么"
- **Human Check 分层**可以直接映射到产品权限体系：免费用户→自动 | 付费订阅→模拟后确认 | 大额操作→多签/人工审批——每一层都在回答"用户需要在什么时候相信 AI，什么时候需要自己看"
- **Revocation** 原则在 BD 合作中同样成立：合作开始的时候，就应该谈好怎么终止。Session Key 到期/额度用完对应合同到期/预算花完——让退出路径从一开始就设计好
- **Simulation** 的"翻译成用户能看懂的语言"是产品设计的核心差异化——很多产品"有模拟"，但模拟结果只有哈希和 gas 估算，用户看不懂就不敢用

## 学习笔记 / Notes

### Agent Wallet 第一性原理

> 控制权不能交给一个概率系统。Agent 只能拿到可验证、可限制、可撤销的行动空间。

- ✅ Agent Wallet 解决的核心矛盾：Agent 可以执行，但只能在用户预先允许的边界里执行
- ❌ 两个极端都不行：Agent 只能给建议不能执行（产品停在"生成方案"），或者权限给得太大（用户不敢把资产交进去）
- ⚠️ BD/产品方向：Agent Wallet 的权限分层理念（Policy/Guard/Revocation）可以直接迁移到**产品权限设计和合作退出条款设计**

### BD/产品场景的权限分层对比

| Agent Wallet 概念 | BD 合作场景 | 活动策划场景 |
|-----------------|------------|------------|
| Session Key | 合作伙伴的限时授权 | 志愿者临时后台账号 |
| Policy | 合作金额上限、范围、时间 | 活动预算上限、参与人数上限 |
| Guard | 自动拦截超预算申请 | 自动拒绝超额报名/重复报名 |
| Simulation | 合同签署前模拟对账 | 活动前模拟报名情况/流水试算 |
| Human Check | 大额支付双人审批 | 大额赞助商需要人工审核 |
| Revocation | 合作到期自动终止 | 活动结束后权限自动回收 |

📚 **Day 11 — Agent Wallet: Permission Boundaries for AI Agents**

Today's focus is Agent Wallet — not about giving an AI a private key, but about defining boundaries that are verifiable, limited, and revocable. Without this layer, every AI×Web3 project faces a dilemma: either the AI can't execute anything (stuck at "generating a plan"), or it has too much power and users won't trust it with real assets.

Key concepts:
- **Session Key** — a temporary, limited key for Agent actions. Not a "mini private key" but a bundle of constrained capabilities (time, amount, targets, revocability).
- **Policy + Guard** — the Policy says what the Agent can do in machine-checkable rules; the Guard deterministically rejects anything that violates those rules before execution.
- **Simulation** — preview transactions before broadcasting, translated into human-readable risk assessment (not just hash + gas).
- **Revocation** — the most important design question isn't "how to authorize" but "how to revoke." Every automated permission needs a visible shutdown button.

From a BD/Product perspective: Agent Wallet's layered permission model (Policy → Guard → Simulation → Human Check) is directly applicable to product permission design. Before launching any AI feature, ask: "What are the boundaries? Who enforces them? How does the user revoke access?" The same framework works for partnership agreements and event planning — define scope, set limits, automate enforcement, and always keep the exit door visible.

**Practice idea:** Design a 3-rule permission policy for your own Learning Agent — what should it auto-execute, what needs your review, and what should it flat-out reject? That's Agent Wallet thinking applied to real products.

> "Giving an AI a wallet is not about trust — it's about boundaries, audit trails, and the ability to say 'stop' at any time." — Day 11 reflection
