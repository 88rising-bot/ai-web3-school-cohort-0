## 今日课程 / Today's Curriculum

📖 **Handbook 章节**: [大语言模型（LLM）](https://aiweb3.school/zh/handbook/ai/llm/) (约 6 分钟阅读)

### 核心知识节点

**1. Token**
- 模型处理文本的基本单位，不等于汉字/英文单词
- 影响：上下文容量、调用成本、关键信息的可见性
- 代码/JSON/表格比普通文本更吃 token

**2. Embedding**
- 将文本/代码映射成向量，衡量语义相似度
- 适用：搜索、聚类、推荐、RAG
- 注意：向量相似度 ≠ 事实正确性，不能替代来源校验

**3. Transformer**
- 核心能力来自 Attention — 关注输入中的不同位置
- 工程事实：擅长找模式，但不等于稳定数据库
- 可能因上下文缺失或相似模式误导产生错误归纳

**4. Hallucination（幻觉）**
- 模型生成看似合理但不真实的内容
- 在问答场景 = 质量问题；在执行场景 = 流程风险
- 错误参数、错误权限解释可能进入自动化链路

**5. Multimodal**
- 模型处理文本+图片+代码等多种输入的能力
- 决定产品能接收什么类型的信息

### 在 AI x Web3 中的位置

> "LLM 可以帮你理解、归纳、生成和规划，但不能单独承担事实来源、权限判断和最终执行。"

**BD/产品视角的关键思考：**
- LLM 是能力底座，不是数据库 — 需要外部事实校验
- 越靠近真实动作（交易、签名、支付），越需要确定性规则
- Agent 工作流中，LLM 负责理解意图和生成计划，钱包和合约负责执行边界

## 学习笔记 / Notes

### LLM 第一性原理

> "LLM 的核心是 next-token prediction。它不存储知识，而是在上下文里找模式。"

- ✅ LLM 擅长：总结、翻译、改写、结构化输出、初步分析
- ❌ LLM 不能：保证事实准确、执行权限判断、替代确定性规则
- ⚠️ 对 BD 来说：AI 提供的信息需要二次验证（链上数据、官方文档）

### Token 的工程影响

| 场景 | Token 消耗 | 建议 |
|------|-----------|------|
| 中文长文 | 中等 | 关键信息摘要前置 |
| 长 JSON/合约代码 | 高 | 只传必要字段 |
| 多轮对话 | 累积 | 定期清理历史 |
| 混合中英 | 较高 | 尽量单语言表述 |

### Hallucination 风险地图

```
问答场景 → 答案不准确 → 可以人工复核
执行场景 → 错误参数/合约调用 → 资产损失风险
```

### BD 视角的 Prompt 实践思路

一个好的 Prompt 结构：
1. **角色定义** — "你是一个 Web3 BD researcher"
2. **任务说明** — "分析这个协议的合作伙伴生态"
3. **边界约束** — "只使用我提供的链接和数据，不要编造"
4. **输出格式** — "输出为表格：项目名 / 合作类型 / 时间"
